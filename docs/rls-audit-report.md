# RLS Audit Report — Candella

Generated by tracing every `sb.from()` / `sb.rpc()` call in the codebase
against the page/auth context it runs in. Nothing here was guessed —
every claim below cites the file and line it came from. No frontend code,
checkout flow, stock logic, ID system, or DB schema was changed as part of
this audit (the SQL file has one clearly-flagged exception, explained in
its own section below).

## Current Risk Level: 🔴 CRITICAL

RLS is disabled on all 5 tables with zero policies. Right now, **any
person with the public anon key (which is plaintext in
`js/config/supabase.js`, by design) can read and write every row in every
table directly via the Supabase REST API**, completely bypassing the
app's own UI and logic. This includes:
- Reading every customer's name, phone, and address from `orders`.
- Reading employee salary/allowance data from `settings`
  (`monthlyClosing_*` keys).
- Inserting, editing, or deleting any `product`.
- Editing or deleting any `admins` row (privilege escalation: insert a row
  with `role: 'super'` pointing at any auth UID you control, and you own
  the dashboard).
- Setting any order's status directly, including marking an unpaid order
  `delivered`, or un-cancelling a cancelled one.

This isn't a "could theoretically happen" risk — every one of these is a
direct, unauthenticated REST call away with the existing anon key. **This
must be fixed before launch.**

## Task 2 — Minimum Required Access (least privilege)

- **admins**: a logged-in user may see their own row (needed for the app
  to know "am I an admin, and what role"). Only a *super* admin may see
  every row, or edit/delete one. No one gets INSERT via the app.
- **products**: anyone may read (it's a public storefront). Only an admin
  (any row in `admins`, regardless of role — `products.js` never checks
  `role`) may insert/update/delete.
- **reviews**: anyone may read and insert (that's the entire feature —
  no login wall exists today). No one — not even admins — may update or
  delete, because that capability doesn't exist in the app.
- **orders**: anyone may insert (guest checkout) and read (track-by-
  number, by design, with no other credential). Anyone may update *only*
  to self-cancel a not-yet-finalized order. Only an admin may otherwise
  change status, and only an admin may read with the intent of managing
  (the same broad read policy serves both purposes since there's no
  separate "customer" identity to scope by).
- **settings**: anyone may read public-content keys. Only an admin may
  read payroll keys or write anything at all.

## Task 3/4 — Policies & SQL

See [docs/rls-policies.sql](rls-policies.sql) for the complete,
ready-to-run SQL. Each policy block has an inline comment explaining
exactly which code path it exists for and why it's shaped that way.
Highlights not obvious from the matrix alone:

- **`orders` UPDATE is split into two policies** — a public one that can
  only flip an active order to `cancelled` (and can't touch a row that's
  already cancelled/delivered/shipped), and an admin one with full access.
  A `GRANT UPDATE (status, cancel_reason)` on top of the public policy
  blocks a hand-crafted request from sneaking other column changes into
  the same call.
- **`orders` INSERT requires `status = 'new'`** — matches the fact that
  every real checkout insert already hardcodes this value
  (`checkout.js`), so legitimate traffic is unaffected.
- **`settings` SELECT excludes `monthlyClosing_*` and `customCosts`** from
  the public policy — these contain employee payroll and internal cost
  data respectively, and nothing in the app reads either from a
  public-facing page, so there's no behavior to preserve by exposing them.
- **The stock RPCs need to be `SECURITY DEFINER`** for checkout to keep
  working at all once `products`/`settings` are locked down — the SQL
  file includes a verification query to check this *before* assuming
  it's already true, plus the (commented-out, fill-in-the-blank) fix if
  it isn't. This is the one place this pass touches anything
  schema-adjacent, and only the function's *privilege mode*, never its
  logic.

## Task 5 — Risk Analysis

### What RLS fixes
Every item listed under "Current Risk Level" above — direct unauthenticated
read/write access to every table — goes away. An attacker with only the
anon key can no longer list all orders, read payroll data, plant a fake
"super" admin, or edit products without legitimately being an admin.

### What RLS does **not** fix (pre-existing, out of scope to change here)

1. **🔴 Critical: the 4-digit `short_id` used for order tracking has only
   9,000 possible values** (`js/services/orderService.js`,
   `generateShortId`). Since `orders` SELECT must stay public for the
   Track Order page to work at all, **anyone can brute-force or simply
   guess a 4-digit code and read another customer's full name, phone, and
   address.** RLS cannot fix this without breaking the track feature —
   the actual fix would be a longer/higher-entropy order code, or pairing
   the lookup with a second factor (e.g. phone number must also match).
   That's a product/ID-system change, explicitly out of scope for this
   pass — flagging it as the most important *follow-up* item.
2. **Cost data inside `settings.giftSet` is visible to anyone who opens
   browser dev tools**, even though the UI never displays it. The
   `giftSet` row must stay publicly readable (the Custom Builder reads
   prices/images from it with no login), and RLS operates at the row
   level, not the JSON-field level — it can't selectively hide the `cost`
   key inside an otherwise-public row. Not fixable without splitting
   public display data from internal cost data into separate
   tables/columns, which is a schema change outside this pass's scope.
3. **The "password required before risky status changes" protection in
   `dashboard.js` is frontend-only.** Both `super` and `custom` admins
   write to `orders` using their own already-authenticated session — the
   password prompt re-verifies the *super admin's* password using a
   disposable client, but the actual database write still happens as
   whichever employee is logged in. The new `orders_admin_full_update`
   policy correctly mirrors this (any admin, not just super, can update),
   which preserves current behavior — but means the extra password check
   is a UX safeguard, not a real authorization boundary at the DB level.
4. **The legacy stock fallback path in `stockService.js`
   (`adjustCustomGiftStock`'s non-stable-id branch) will stop working**
   once `settings` is locked down for anonymous writes. The code's own
   comments already call this an accepted tradeoff ("none currently exist
   in the live database") — confirmed during this audit, not newly
   discovered, but worth restating: if there is ever an old in-flight
   order from before the stable-id system, cancelling it will no longer
   restore its stock automatically.

## Deployment Order

**Step 1 — Backup**
Take a Supabase database backup (Dashboard → Database → Backups, or
`pg_dump` if you have direct access) before touching anything. RLS
mistakes can lock you out of your own data through the app instantly.

**Step 2 — Verify RPC security mode**
Run only the `SELECT proname, prosecdef ...` query at the bottom of
`rls-policies.sql` *first, by itself*. If both stock RPCs already show
`true`, skip the ALTER statements. If either shows `false`, fix that
specific one before proceeding — this is the one step that will silently
break checkout if skipped.

**Step 3 — Run the policy SQL in a staging/test project if you have one**
If not, proceed directly but be ready to roll back (see Step 6).

**Step 4 — Run `rls-policies.sql` in full** via the Supabase SQL Editor.

**Step 5 — Manually test every flow in the Verification table above**,
in this order, on the live site:
1. Place a normal product order through checkout → confirm it succeeds
   and stock decreases.
2. Place a Custom Gift Set order → confirm it succeeds and candle/
   container/accessory stock decreases.
3. Track that order by its 4-digit code → confirm it loads.
4. Cancel it from the track page → confirm stock restores.
5. Log into the dashboard as a regular ('custom') employee → confirm
   order list loads, status update works.
6. Log into the dashboard as the super admin → confirm employee list
   loads, Monthly Closing loads.
7. Submit a review from the home page → confirm it appears.
8. Open the Home Editor → confirm catalog read/save still works.

**Step 6 — If anything in Step 5 fails**, the fastest rollback is
`ALTER TABLE <table> DISABLE ROW LEVEL SECURITY;` on just the affected
table while you debug — this re-opens that one table without undoing the
others.

**Step 7 — Only once every item in Step 5 passes**, consider this safe for
production traffic.
